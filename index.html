多线程

为什么使用多线程?

通过减少CPU空闲的时间来达到提高效率的目的。

进程与线程

进程:指一个内存中运行的应用程序，每个进程都有自己独立的一块内存空间，并且一个进程可以启动多个线程。比如在Android系统中一个运行一个apk就相当于启动了一个进程。

线程:指进程中的一个执行流程。线程总是属于某个进程，进程中的多个线程共享进程的内存。
“同时”执行是人的感觉，线程之间实际上轮换执行。

Java中的线程

1.Thread类

2.Runnable接口

线程中的一些常见问题

1.线程的名字可以开发者自己设置，也可以是JVM给定。
2.通过Thread.currentThread()获取线程的名称，且只能获取当前线程名称。
3.一系列线程按照一定的顺序启动，但不能保证按照该顺序执行完成。
4.当线程run方法结束时该线程完成。
5.线程只能启动一次，多次启动回报java.lang.IllegalThreadStateException:Thread already started。
6.在一个CPU的机器上，实际上在同一时刻只能运行一个线程。
7.尽管我们没有办法控制线程调度，但我们可以通过别的方式来影响线程调度。

线程状态转换


1.新状态:new了线程对象，但是没有调用start方法。
2.可运行状态:线程有资格运行，但是调度还没有把他选定为运行状态时所处的状态。此时该线程已经调用的start方法。
3.运行状态:线程调运从运行池中选定一个线程作为当前线程时所处的状态。线程唯一进入运行状态的方式。
4.等待/阻塞/睡眠状态:线程没有条件运行，如果某件事件出现会返回到可运行状态。
sleep()<–>sleep()结束
yield()<–>再次获取CPU
wait() <–> notify()/notifyAll()
join() <–> join()中断 直到加入的线程执行完之后，当前线程才能继续执行
synchronized/lock<–>同步锁被释放
上述几种情况可导致运行状态到阻塞、阻塞到可运行状态或运行状态
5.死亡状态:当线程的run方法完成就认为它死去。这个线程对象也许是活的，但是他已经不是一个单独执行的线程。

影响线程调度

1.线程的交互
wait()/wait(long timeout)/wait(long timeout,int nanos)
当前线程等待，直到其他线程调用此对象的notify()或notifyAll()或超过指定的时间量
notify()/notifyAll()
唤醒此对象上等待的单个(所有)线程
注意:上述这些方法必须在同步环境中调用。换句话说线程不能调用对象上等待或通知的方法，除非该线程拥有此对象的锁。
2.线程的休眠
sleep(long millis)/sleep(long millis,int nanos)
当前线程休眠，使该线程让出CPU资源给其他线程，被休眠线程的休眠时间大于或等于mills
3.线程的优先级
setPriority(int priority) 0~10
优先级高的线程获取CPU资源的概率更大。
4.线程的让步
yield()
当前运行的线程让出CPU资源，并且该线程变为可运行状态，线程调度可以再一次选中该线程
5.线程的合并
join()/join(long mills)/join(long mills,int nanos)
一个线程必须等待另一个线程执行完毕才能执行。(等待时间最长为mills/mills+nanos)

线程的同步

为什么会有同步问题的提出？
当多个线程访问同一个数据对象时，会对数据造成破快。 列子。
通过锁的方式保证数据的合理性。
Java中内置的锁

1.synchronized
注意:释放锁就是指持锁的线程退出了synchronized同步的方法或代码块

5.原子量 AtomicLong(XXX)
注意:原子量只能保证单个变量在某一个操作过程是安全的，无法保证在整个代码块的安全性， 所以还是需要Lock/synchronized来保证整个代码块的安全性。

小结
1.当一个类能很好的同步来保护它的数据时，我们称这个类为“线程安全的”。
2.线程同步的目的是保护多个线程操作数据时不被破坏。
3.锁同步的是方法，不是对象或者变量。
4.线程同步是通过锁来实现，每个对象有且只有一个锁,一旦某一线程获取了该锁，其他访问该对象的线程就不能再访问该对象的同步方法(发送阻塞)。
5.死锁是线程间相互等待锁造成的，在开发过程中需要注意。

线程池

为什么会出现线程池，这是什么鬼？
执行一个线程所需要的时间 = 线程创建的时间 + run方法执行的时间(包括同步的时间) + 线程销毁的时间
线程池的思想就是开辟一块内存空间，里面存放众多(未死亡)的线程，当有线程任务时，从池中直接取出一个，该任务执行完之后再放入池中，这样可以避免反复创建和销毁线程带来的性能开销，节省系统资源。

各种类型的线程池
1.newCachedThreadPool()

可缓存线程池，先查看池中有没有前创建好的线程，有，复用，没有，创建新的线程加入池中。
注意:可复用的线程必须是timeout idle内的池中线程，缺省是60s,超过这个idle时长，线程实例会被终止并移出池。所以该类型的线程池适用于一些生存周期比较短的任务。

2.newFixedThreadPool()

固定大小的线程池，也能复用线程，但不能随时创建新线程，在任意时间点，最多只有固定数量的活动线程，如果有新的线程要创建，只能放到另外的队列中等待，直到当前线程中的某个线程终止直接被移出池子。
注意:该类型的线程池没有idle,创建的线程会一直存在。

3.newScheduledThreadPool()

调度型线程池，线程池中的线程数也是固定的，但是可以根据schedule一次delay执行，换句话可周期性执行

4.SingleThreadExecutor

单任务线程池，任意时间线程池中只有一个线程

5.ThreadPoolExecutor

自定义线程池

corePoolSize:核心大小，线程池初始化的时候，就会有这么大
maximumPoolSize:线程池最大线程数
keepAliveTime:如果当前线程池中线程数大于corePoolSize,多余的线程，在等待keepAliveTime时间后如果还没有新的线程任务指派给它，它就会被回收。但若将allowCoreThreadTimeOut为true,不论线程池中的线程数是否大于corePoolSize，只要是空闲的线程超过keepAliveTime都会被回收。
unit:超时时间的单位。
workQueue:线程池的任务队列，通过execute方法提交的runnable对象会存储在这个队列中。 默认长度128
threadFactory:线程工厂，为线程池通过创建新线程的功能。
handler:任务无法执行时，回调handler的rejectedExecution方法来通知调用者。

单线程的JavaScript

浏览器的内核是多线程的，它们在内核控制下相互配合以保持同步，一个浏览器中至少会实现三个常驻线程:
javascript引擎线程:javascript引擎是基于事件驱动单线程执行的，JS引擎一直等待着任务队列中任务的到来，然后加以处理，浏览器无论什么时候都只有一个JS线程在运行JS程序。
GUI渲染线程:GUI渲染线程负责渲染浏览器界面，当界面需要重绘（Repaint）或由于某种操作引发回流(reflow)时,该线程就会执行。但需要注意GUI渲染线程与JS引擎是互斥的，当JS引擎执行时GUI线程会被挂起，GUI更新会被保存在一个队列中等到JS引擎空闲时立即被执行。
浏览器事件触发线程:事件触发线程，当一个事件被触发时该线程会把事件添加到待处理队列的队尾，等待JS引擎的处理。这些事件可来自JavaScript引擎当前执行的代码块如setTimeOut、也可来自浏览器内核的其他线程如鼠标点击、AJAX异步请求等，但由于JS的单线程关系所有这些事件都得排队等待JS引擎处理。（当线程中没有执行任何同步代码的前提下才会执行异步代码）

js工作线程实现的方式

1.setTimeout() 使用该函数将任务进行分割
2.Google Gears谷歌开发的浏览器插件工具，不提倡使用。
3.Web Worker借鉴Google Gears，规范定义了一套API，允许 Web 应用创建在后台执行 JavaScript 代码的工作线程。即工作线程在执行过程中不会对主页面造成影响。Web Worker 规范中定义了两类工作线程:

专属工作线程:只能为一个页面所使用,创建专属线程时要指定需要执行的js文件的URL。通过postMessage和onmessage来接收和发送消息。

共享工作线程:可以被多个页面使用。共享工作线程所执行的 JavaScript 代码中，不能使用 onmessage来直接定义接收到消息时的处理方法，而需要使用 onconnect来定义接收到连接时的处理逻辑

Android动画

Android 帧动画

概念
帧动画非常容易理解，就是由N张静态图片收集起来，然后通过我们的控制依次显示这些图片。
两种用法
1.采用XML文件实现
2.Java代码实现(通过类AnimationDrawable)
具体实现方式在这就不详细说，百度一大堆~

Android补间动画(视图动画)

Android补间动画提供了AlphaAnimation(透明),RotateAnimation(旋转),TranslateAnimation(平移),ScaleAnimation(缩放)四种动画方式以及提供了AnimationSet动画集合，混合使用多种动画。
具体实现请参考百度~~

Android属性动画(Android3.0之后出现)

为什么会出现属性动画？
1.补间动画功能比较单调，只有四种动画(透明度，旋转，缩放，平移)。
2.补间动画针对的对象只是UI控件。
3.补间动画改变的只是显示效果,不会去改变View的属性。
eg:当某元素发生了补间动画之后，该元素的点击事件还在原来的位置。
什么是属性动画？
1.属性动画是补间动画的增强版，不止可以实现四种动画效果，还可以定义任何属性的变化。
2.执行动画的对象不只是UI控件，可以对任何对象执行动画。
3.属性动画通过对目标对象进行复制来修改其属性,完美解决了补间动画的缺陷(只是显示效果,不会去改变View的属性)。
相关API
1.Animator
创建属性动画的基类，一般不会直接用，一般使用它的两个子类。
2.ValueAnimator
ValueAnimator本身不提供任何动画效果，它更像一个数值发生器，用来产生有一定规律的数字，从而让调用者来控制动画的实现过程。
使用流程
1.调用ValueAnimator的静态方法ofInt(),ofFloat(),或ofObject()创建ValueAnimator的实例。
2.调用实例的setXXX方法设置动画持续时间 ，插值方式，重复次数等。
3.调用实例的addUpdateListener添加AnimatorUpdateListener监听器，在该监听器中可以获取ValueAnimator计算出来的值，可以将这些值应用到指定对象上。
4.调用实例的start方法开启动画。
具体实例请参考
3.ObjectAnimator
ObjectAnimator，通过该类我们可以直接对任意对象的“任意属性”进行动画操作，“任意属性”中的属性必须有get、set方法。
使用流程
1.调用ObjectAnimator的静态方法ofInt(),ofFloat(),或ofObject()创建ObjectAnimator的实例，与ValueAnimator中的静态方法的区别是，ObjectAnimator中静态方法多了要操作的对象以及对象的属性。
2.调用实例的setXXX方法设置动画的持续时间，插值方式，重复次数等。
3.调用实例的start方法开启动画。
注意:若一个属性没有get、set方法那我们是否就束手无策了呢？当然不，Google提供了两种方案来解决此问题。
方案1：提供自定义一个属性类或者包装类间接的给该属性增加get、set方法。
方案2：提供ValueAnimator实现。

CSS花样选择器

  • 通配符选择器
    用星号“*”类定义，该选择器可以与任何元素匹配
  • 派生选择器
    通过依据元素在其位置的上下文关系来定义样式
  • id选择器
    id选择器可以为标有特定id的HTML元素指定特定的样式。id选择器以 “#” 来定义。
  • 类选择器
    类选择器可以为标有特定class的HTML元素指定特定的样式。类选择器以 “.” 来定义。
  • 元素选择器
    文档的元素是最基本的选择器
  • 属性选择器
    对带有指定属性的HTML元素设置样式
  • 子元素选择器
    为某元素的子元素指定特定的样式
  • 相邻兄弟选择器
    为紧接在另一元素后的元素，且二者有相同父元素指定特定的样式
  • 伪类
    CSS 伪类用于向某些选择器添加特殊的效果

:active 向被激活的元素添加样式
:focus 向拥有焦点的元素添加样式

:link 向未被访问的连接添加样式
:visited 向已被访问的连接添加样式
:first-child 向元素的第一个子元素添加样式
:lang 向带有指定lang属性的元素添加样式
  • 伪元素
    与伪类类似也是向某些选择器设置特殊效果
属性 描述
:first-letter 向文本的第一个字母添加特殊样式
:first-line 向文本的首行添加特殊样式
:before 在元素之前添加内容
:after 在元素之后添加内容

Android性能优化

##why
大部分手机的屏幕刷新频率是60hz,若在1000/60 = 16.67ms不能完成一帧的任务，就会造成丢帧现象，用户就会感受到卡顿。

Android UI CPU GPU

CPU负责把UI组件计算成Ploygons、Texture,然后交给GPU进行栅格化渲染。

1.去除非必需组件,非必需组件会造成CPU的重复计算以及GPU的额外资源

2.减少GPU的Overdraw(过度绘制:屏幕上的某一个像素在同一帧的时间内被渲染多次)
检查过度渲染工具:手机设置->开发者选项->ShowGPUOverdraw的选项，可以观察到UI的Overdraw情况。


蓝、绿、淡红、深红代表了4种不同程度的Overdraw情况。

优化注意点:

移除Window默认的Background

移除XML布局文件中不必要的Background

按需显示占位图片

ClipRect && Quickreject
Android系统提供了一些组件来避免不可见组件造成的Overdraw，但是对于那些复杂的自定义的View(重写了onDraw方法)，Android系统无法检测具体操作，系统无法监控并优化，也就无法避免Overdraw，针对上述情况我们可以通过canvas.clipRect()来帮助系统识别可见区域。该方法可以指定一块区域，只有在这个区域才会被绘制，其他区域被忽视。我们也可以使用canvas.quickReject()来判断是否没和某个矩形有交集,从而跳过那些非矩形区域的绘制操作。

DisplayList
避免布局的嵌套
Profile GPU Rendering
蓝色:绘制DisplayList的时间
红色:OpenGL渲染DisplayList需要的时间
黄色:CPU等待GPU处理的时间

单例模式

单例模式

定义

确保类中只有一个实例，而且自行实例话并向整个系统提供该实例

场景

避免产生多个对象消耗过多的资源

分类
饿汉单例模式
private static final ClassName name = new ClassName();
public static ClassName getClassName() {
 return name;
}
懒汉单例模式
public class ClassName {
 private static ClassName name;
 private ClassName(){}
 public static syntronized ClassName getClassName() {
  if(name == null) {
      name = new ClassName();
  }
  return name;
 }
}

注意:syntronized关键字，保证在多线程下单例模式对象的唯一性。

优点只有在使用的时候才会被实例化，节约资源。

缺点每次调用都会造成同步开销。

Double Check Lock

既能需要时实例化，又能保证线程安全

public class ClassName {
 private static ClassName name = null;
 private ClassName(){}
 public static ClassName getClassName() {
 if(name == null) {
  syntronized (ClassName.class) {
   if(name == null) {
    name = new ClassName();
   }
  }
 }
 return name;
}

注意:DCL在一定程度上解决了资源消耗、多余的同步、线程安全等问题，但由于java内存模型的原因偶尔会失败(java6版本下使用会出问题)

静态内部类单例模式
public class ClassName {
 private ClassName(){}
 public static ClassName getClassName() {
  ClasssHolder.name;
 }
 // 静态内部类
 private static class ClassHolder{
  private static final ClassName name = new ClassName();
 }
}

注意:保证线程安全以及对象的唯一性，同时也推迟了实例化，推荐使用

枚举单例模式
public class ClassName {}
public enum EnumName {
NAME;
private ClassName name;
EnumName() {
    name = new ClassName();
}
public ClassName getClassName() {
    return name;
}

}

React学习路线

##Step1:React入门

React的基本理解以及注意点

在html中插入react

注意:

（1）jsx与javaScript不兼容,凡是使用jsx都加上type=”text/babel”

（2）首先加载相关的三个库,react.js:React的核心库、react-dom.js:提供与dom相关的功能、browser.js:将jsx转换为javaScript

jsx基本语法规则

遇到html标签(以 < 开头 )，就按照html规则解析，遇到代码块( 以{ 开头 }就用JavaScript规则解析

组件

每个组件都有自己的render方法，并在组件中包含一个顶层标签

PropTypes

组件类的PropTypes属性。校验React组件属性的数据类型。

props

this.props对象的属性与组件属性一一对应。但是注意this.props.children他表示组件的所有子节点，为了避免children数据类型的不确定性带来的问题，react提供了React.Children方法来处理children。

this.ref

通过this.ref获取真实的DOM节点

this.state

一种React与用户交互的状态机制。注意this.props与this.state的区分

组件的生命周期
Mounting:已插入真实的DOM 
Updating:正在被重新渲染
Unmounting:已移除真实的DOM

React 为每个状态都提供了两种处理函数，will 函数在进入状态之前调用，did 函数在进入状态之后调用，三种状态共计五种处理函数。

componentWillMount()
componentDidMount()
componentWillUpdate(object nextProps, object nextState)
componentDidUpdate(object prevProps, object prevState)
componentWillUnmount()

此外，React 还提供两种特殊状态的处理函数

componentWillReceiveProps(object nextProps)：已加载组件收到新的参数时调用
shouldComponentUpdate(object nextProps, object nextState)：组件判断是否重新渲染时调    用
