AndroidHandler

就应用程序而言，Android系统中的java的应用程序和其他系统相同，都是靠消息驱动来工作的，它们大致工作原理如下
1.有一个消息队列，可以往这个消息队列中投递消息
2.有一个消息循环，不断的从消息队列中取出消息，然后处理。
Android Handler 就实现这样的一件事情。提到Android Handler就不得不提到Looper、Message以及它们之间的暧昧关系，下面将揭示它们神秘的面纱。

Handler的消息处理
刚才我们往Looper的消息队列中加入了一个消息，按照Looper的处理规则，他在获取消息后会调用target的dispatchMessage函数，再把这个消息派发给Handler处理，Handler是如何处理消息的呢？

dispatchMessage定义了一套消息处理的优先级机制，它们分别是:

  1. Message如果自带callback，则交给callback处理
  2. Handler如果设置的全局的mCallback,则交给mCallback处理
  3. 如果上述都没有，该消息则会被交给Handler子类实现的handleMessage处理。当然这需要从Handler派生并重写handleMessage

Android机型适配

注:Android设备兼容包括:语言适配、屏幕适配、不同系统版本适配

语言适配

若决定支持某种语言，则需要创建资源子目录和字符串资源文件
eg:
MyProject/
—res/
——values/
———strings.xml
——values-es/ 西班牙语
———strings.xml
——values-fr/ 法语
———strings.xml
添加不同区域语言的字符串到相应的文件

屏幕适配

4种普通尺寸:小(small)、普通(normal)、大(large)、超大(xlarge)
4种普通分辨率:低精度(ldpi)、中精度(mdpi)、高精度(hdpi)、超高精度(xhdpi)

屏幕的方向也是一种考虑适配的尺寸变化
解决方案:创建不同的layout
eg:
MyProject/
—res/
——layout/
———main.xml
——layout-large/
———main.xml
——layout-land/
———main.xml
——layout-large-land/
———main.xml

注意:Android3.2及以上版本支持定义屏幕尺寸的高级方法，它允许根据屏幕的长度和宽度指定layout

适配不同屏幕的注意点

  1. 使用wrap_content、match_parent尺寸值代替硬编码的尺寸
  2. 尽可能多的使用RelativeLayout,精准得到layout与子控件的关系
  3. 使用最小宽度限定词 (最小限定词允许你根据设备的最小宽度(dp单位)来指定不同布局)
    注意:最小宽度限定词只能在Android3.2或者更高版本使用

  4. 使用布局别名
    eg:
    res/layout/main.xml
    res/layout-large/main.xml
    res/layout-sw600dp/main.xml
    最后两个文件是一样的,layout-sw600dp适配的是Android3.2的设备,layout-large适配的是其他低版本的Android设备,为了避免重复以及后期维护的复杂性采用别名方式进行适配
    eg:
    res/layout/main.xml
    res/layout/main_two_pans.xml
    然后添加这两个文件:
    res/values-large/layout.xml

    res/values-sw600dp/layout.xml

  5. 背景图使用.9图

  6. 使用密度独立像素(dp、sp)的单位
  7. 提供不同分辨率的图片

适配不同系统版本

  1. 指定最小和目标API级别
  2. 运行时检查系统版本
    Android在Build常类中提供了对每一个版本的唯一代号，我们可以利用这些代号在我们的app中建立条件，保证依赖于高级别的API代码
  3. 使用平台风格和主题

Buck

Android Buck支持Mac os x 、Linux 、 Windows
IOS Buck只支持Mac os x、Linux 、 Windows

Mac OS X下构建Android

搭建Buck环境

1.Homebrew方式下载Buck

$ brew update
$ brew tap facebook/fb
$ brew install --HEAD buck

2.手动方式下载Buck

手动构建Buck的前提要下载以下工具:
Command Line Tools
Oracle Java Development Kit
Apache Ant 1.9 (or newer)
Python 2.7
Git
Watchman

你可以选择使用Homebrew安装上面的工具.

上面工具安装成后可以按照下面的方式 install buck
git clone https://github.com/facebook/buck.git
cd buck
ant
./bin/buck –help

build安装成功界面

配置环境变量

$ cd ~
$ vim ~/.bash_profile
添加如下配置
export PATH=$HOME/buck/bin:$PATH
使配置立即生效
$ source ~/.bash_profile

Buck构建Android

  1. 快速构建
    $ touch .buckconfig && buck quickstart

    该命令在执行过程中会要求补全两个参数的值:
    –dest-dir: 生成的Android工程的目录
    –android-sdk: 本地Android SDK的目录
    注意:新版本–android-sdk参数失效了,可以在开发平台手动设置SDK

    Buck构建Android的目录结构

    进入到项目根目录执行如下命令

    编译应用程序
    $ buck build android别名
    安装应用到设备
    $ buck install –run android别名

    创建IntelliJ工程
    $ buck project –ide IntelliJ

Buck的基本理解

Build Rule

构建规则是从一组输入文件中生成一组输出文件的过程，是一种泛称。Buck已经内建了一系列的构建规则。比如构建Android代码是会用到规则

android_binary()生成一个apk文件，android_library() 生成jar包等

每个构建规则至少包括三个参数

1.name:构建规则的名称，在一个构建文件中必须保持其唯一性

2.deps:构建规则的依赖项，以一个构建目标的列表的形式表示

3.visibility:可见性，表明其他构建规则是否可以依赖该规则构建，以一个构建目标模式的列表形式表示

示例1

android_resource(
  name = 'res',
  res = 'res',
  assets = 'assets',
)

android_library(
  name = 'src',
  srcs = glob(['src/**/*.java']),
  deps = [
    ':res',
  ],)

android_binary(
  name = 'hello',
  manifest = 'AndroidManifest.xml',
  keystore = '//keystores:prod',
  package_type = 'release',
  proguard_config = 'proguard.cfg',
  deps = [
    ':res',
    ':src',
  ],)

注意:buck中构建规则依赖的规则，会先于构建规则本身进行构建，构建规则和它的依赖之间是一个有向图，并且这个图的无闭环的。

Build Target

Build Target 构建目标是一个字符串，用于标识工程中的某个构建规则，它常常作为Build Rule的参数使用。

'//java/com/example/activity:activity'

主要有三部分组成:

1.// 前缀表示路径是相对于工程的根目录

2.java/com/example/activity表示构建文件buck位于该目录中

3.:后面的activity表示构建文件里面定义的构建规则的名称，在一个构建文件中，构建规则的名字是唯一的

在同一个构建文件中，可以使用构建目标的相对路径来引用。相对路径以(:)冒号开始，后面紧跟三大组成部分的第三部分。如示例1

Build File

Build File 构建文件用于定义一个或者过个构建规则，统一命名为Buck.在一个工程中的java源文件只能被离它最近的Buck文件引用。

示例

java/com/facebook/base/BUCK
java/com/facebook/common/BUCK
java/com/facebook/common/collect/BUCK

base/Buck文件中的构建规则只能引用base/目录下的源文件

common/Buck文件中的构建规则可以引用该目录下除去子目录collect之外的所有源文件

collect/Buck文件中的构建规则只能引用collect中的源文件

那么我们如何在common中引用base和collect中的构建规则呢?答案是 使用deps引用这些文件

示例

java/com/facebook/common/test/的代码需要依赖java/com/facebook/common/collect/包下面的代码,假设java/com/facebook/common/collect/BUCK文件中有一个构建规则如下：

java_library(
   name = 'collect',
   srcs = glob(['*.java']),
   deps = [
 '//java/com/facebook/base:base',
  ],)

那么java/com/facebook/common/BUCK文件应该定义规则如下：

java_library(
   name = 'test',
  srcs = glob(['test/*.java']),
  deps = [
        '//java/com/facebook/base:base',
    '//java/com/facebook/common/collect:collect',
  ],)

Build Target Pattern

Build Target Pattern构建目标模式是一个用来匹配一个或者多个构建目标的字符串，一般作为参数传递给构建规则中的visibility属性。一个构建目标其实也是一个构建模式，只是它匹配到的是它自己.

# 匹配到 '//apps/myapp:app'
'//apps/myapp:app'

以冒号:结尾的构建目标模式可以匹配到同一目录中的构建目标

# 匹配到 '//apps/myapp:app_debug' 和'//apps/myapp:app_release'
'//apps/myapp:'

以/…结尾的构建目标模式可以匹配到该目录及其子目录中的构建目标：

# 匹配到 '//apps:common' and '//apps/myapp:app'
'//apps/...'

RN_Pad端优化

本篇文章是介绍在实践开发中遇到的一些性能问题给出的优化方案。

ListView的优化

  1. 注意rowHasChanged函数中条件的设置，如果设置不当会造成全量渲染导致卡顿现象。

  2. 设置ListView的initialListSize属性，指定组件刚挂载的时候渲染N行，来确保首屏显示合适数量的数据。

  3. ListView进行分页显示.

ReactNativeRouterFlux的优化

  1. 设置duration={0},取消页面之间的跳转动画，消除跳转页面卡顿现象。

优化首次登陆时，融云推送大批量消息时的逻辑

  1. 当首次登陆接收融云大批量消息时,将js与原生端的推送消息改为批量推送,减少js端与原生端连接的性能损耗。

  2. js端接收到消息后的处理逻辑,一次性设置到Redcer中,消除患者沟通列表的多次渲染。

数据请求时的处理

  1. 为所有有网络请求的页面添加loading效果,提升用户体验,消除不连贯的体验。

HTTP Request耗时日志

  1. 去掉HTTP Request耗时日志,这是由于react-native本身框架的原因造成。

录音功能

  1. 正在录音中可以返回到其他页面做别的任务,也可以黑屏或者后台录音。

  2. 修改录音绑定和workbook处开始录音的处理逻辑,一次只能有一个录音。

其他页面的优化

  1. 无档案患者列表, 数据结构与新UI所需要的数据结构不符,转换数据结构。

  2. 患者动态页面, 数据结构与新UI所需要的数据结构不符,转换数据结构。
